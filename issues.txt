This is an excellent question! While Python is indeed platform-independent and translates newlines automatically when reading local files using open('file.txt', 'r') (called Universal Newlines), it does not translate newlines for strings fetched from the network, APIs, or database records.

Here are the scenarios where you will encounter \r\n or \r in this RAG application:
When you press Enter, you don't actually insert a visible blank line. The OS inserts one or more special characters.


Scenario 1: API and Network JSON Payloads
The documents in this system are loaded from remote connectors:

The GitHub API returns base64-encoded file text. (A Base64-encoded file is simply the file's raw bytes converted into a text representation using the Base64 encoding scheme, so it can be safely stored or transmitted in systems that expect text. It is not encrypted, not compressed, and not hashed—it's just a reversible encoding.)
The Confluence REST API returns HTML/markdown page contents.
The Google Drive API downloads raw documents.
When Python reads an API response (e.g. response.json()), the payload is loaded as a raw string. Python does not perform automatic newline translation on string data received over HTTP/JSON. If a Windows developer originally committed a file to GitHub with \r\n, the string fetched via the GitHub API will contain literal \r\n characters.

Scenario 2: Text Uploaded via HTTP POST Requests
If a user copies a chunk of text on their Windows PC and pastes it into the chat frontend, or uploads a file via a multipart form request, the HTTP request body containing the text will preserve the Windows line breaks (\r\n). When FastAPI parses the request body into your Python variable, it remains \r\n.

Scenario 3: Byte Decoding
If a connector downloads a document as binary data and decodes it manually:

python
raw_bytes = response.content  # Raw bytes from HTTP
text = raw_bytes.decode("utf-8")  # No newline translation occurs here!
The resulting string will preserve the exact newline characters (\r\n or \r) from the source bytes.

Why is this critical for the RAG Pipeline?
Dangling Carriage Returns (\r): If we split the text on \n (line 9: text.split("\n")) without replacing \r\n first, the carriage return \r character gets left behind at the end of every single line string. E.g., ["Hello world\r", "This is line two\r"]. This messes up word splits, formatting, and token count calculations.
SHA-256 Hash Matching: The system calculates the document hash (hash_text) to detect if a file has changed before re-indexing. If a developer edits a file on a Mac (\n) and another edits it on Windows (\r\n), the text is visually the same, but the SHA-256 hash would be completely different because of the hidden \r bytes. Normalizing them ensures the exact same hash regardless of the operating system that authored the file.

# Understanding UTF-8 and Why SHA-256 Needs It

## What is UTF-8?

**UTF-8 is an encoding.**

An **encoding** is simply a set of rules that tells the computer:

> **"How do I convert human-readable characters into bytes (numbers)?"**

Computers do **not** understand letters like `A`, `B`, or `Hello`.

They only understand **bits (0s and 1s)**.

So before storing or processing text, the text must be converted into numbers.

---

## Example

Suppose we have the string:

```python
text = "Hello"
```

This is a Python **string** (`str`).

Internally, it represents the characters:

```text
'H'  'e'  'l'  'l'  'o'
```

But SHA-256 cannot work with characters.

It only works with **bytes**.

So we convert the string into bytes using UTF-8.

```python
b = text.encode("utf-8")
```

UTF-8 converts each character into its byte representation.

```text
H → 72
e → 101
l → 108
l → 108
o → 111
```

So internally, the data is:

```text
[72, 101, 108, 108, 111]
```

Each number is **one byte** (8 bits).

---

## Then why does Python print

```python
print(b)
```

Output:

```python
b'Hello'
```

instead of

```text
[72, 101, 108, 108, 111]
```

Because Python is trying to be human-friendly.

Those byte values (72, 101, 108, 108, 111) happen to correspond to printable ASCII characters.

So instead of displaying:

```text
72 101 108 108 111
```

Python displays them as:

```python
b'Hello'
```

**Important:** This is only a display format.

Internally, they are still bytes (numbers).

You can verify this:

```python
b = "Hello".encode("utf-8")

print(list(b))
```

Output:

```python
[72, 101, 108, 108, 111]
```

Or:

```python
print(b[0])
```

Output:

```python
72
```

So `bytes` is essentially a sequence (array/vector) of numbers between **0 and 255**.

---

## What does SHA-256 need?

SHA-256 is a mathematical algorithm.

It **does not understand text or characters.**

It only understands a **sequence of bytes**.

So this:

```python
"Hello"
```

must first become:

```text
[72, 101, 108, 108, 111]
```

before SHA-256 can process it.

You can mentally think of SHA-256 as receiving:

```text
SHA256([72, 101, 108, 108, 111])
```

(Internally, it processes the bytes themselves, not a Python list, but this is an accurate mental model.)

---

## Complete Flow

```text
Human Text
    │
    ▼
"Hello"
    │
    ▼
Python String (Unicode)
    │
    ▼
encode("utf-8")
    │
    ▼
Bytes
[72, 101, 108, 108, 111]
    │
    ▼
Stored as bits / byte array
01001000 01100101 01101100 01101100 01101111
    │
    ▼
SHA-256
    │
    ▼
256-bit Hash
    │
    ▼
Hexadecimal String
185f8db32271fe25f561a6fc938b2e264306ec304eda518007d1764826381969
```

## Easy way to remember

* **UTF-8** = Converts **text → bytes (numbers)**.
* **`encode("utf-8")`** = Performs that conversion.
* **`bytes`** = A sequence (array/vector) of numbers from **0 to 255**.
* **`b'Hello'`** = Python's human-friendly way of displaying those bytes.
* **SHA-256** = Accepts only **bytes**, not strings, and computes a fixed-size hash from them.

Why do we use it? (Context Management)
LLMs have a strict limit on how much text they can read at once (the "context window").

To prevent overloading the LLM, the RAG engine needs to monitor how large each document chunk is.
Instead of installing a heavy tokenizer library (like OpenAI's tiktoken or Hugging Face's transformers which would make the backend slow to load and install), the code uses word count as a proxy.
Rule of thumb in English: 1 word is roughly equal to 1.3 to 1.4 tokens. Counting words gives a fast, lightweight estimate that is perfectly fine for basic chunk sizing.

LLM does not see word by word, but token by token

Word2Vec is excellent at representing individual words, but RAG needs to represent the meaning of an entire chunk of text.

Word2Vec is an obsolete technology (from 2013) that cannot be used for RAG:

Word2Vec only embeds single words: Word2Vec has a static vector for "bank" and a static vector for "river". 
    To get the vector of a sentence, you have to mathematically average the words together. This completely destroys syntax, grammar, word order, and context (e.g.,
     "not good" and "good, not" would have the exact same averaged vector).

Sentence Transformers embed context: This is a transformer-based encoder (based on BERT). It reads the entire
     text chunk at once, captures how words interact with each other in context, and produces a single vector representing the semantic meaning of the whole block of text.
(open soruce model, size: 90 MB)
Compute: Because the model is so small, it computes vectors in milliseconds using just your CPU. You do not need a GPU.

source_system_id:
What it is: The database integer ID of the connector platform (e.g., 1 for GitHub, 2 for Confluence, 3 for Google Drive).
Why it's a Foreign Key (FK): It links directly to the source_systems table. This prevents bugs by ensuring you cannot insert a document belonging to a non-existent connector.

external_doc_id:
What it is: The unique string ID that the external platform uses to identify that file.
Because we don't control GitHub or Google Drive, we cannot use simple database auto-increment numbers (1, 2, 3) to identify their files. We must store their native ID strings.
Examples:
Google Drive: The file's GUID (e.g., '1A2b3c4d5e...').
GitHub: A composite string containing the path (e.g., 'github:company/auth-service/docs/README.md').

1. What is on_conflict?
In standard SQL databases, if you try to INSERT a row that duplicates an existing UNIQUE constraint, the database throws a Duplicate Key Error and crashes the request.

on_conflict="source_system_id,external_doc_id" tells PostgreSQL:

"If you try to write this document, and you find a row that already has the exact same combination of source_system_id and external_doc_id, do not crash. Instead, update that existing row's columns with this new payload."

1. What is a "Resource Scope"? (The Concept)
Think of a Resource Scope as a data folder or security container.

In an enterprise company, you don't just dump all files into one big pile. Instead, files are organized into containers:

A GitHub repository named payment-gateway
A Google Drive folder named Q3-Finances
A Confluence space named HR-Onboarding
Each of these containers is a Resource Scope. In our database, this table maps the container to its security rules (e.g., "Only people in the TECH department with L2 clearance can search the payment-gateway scope").

2. Why it CANNOT be system-seeded
System-seeded tables (like departments or source_systems) are static. A company rarely adds a new department, and your codebase rarely adds a new connector type.
resource_scopes is dynamic because employees create new folders, repositories, and spaces every day.
If a manager creates a new Google Drive folder tomorrow called Marketing-Specs-2026, your backend needs to represent that new folder in the database.
How does it get populated?
Via Admin Endpoints: An administrator creates a new scope using the API:
http
POST /admin/resource-scopes
Payload: { "name": "Payment Service", "external_resource_id": "github:company/payment-service", ... }
Auto-Discovery during Ingestion: When a sync connector runs and scans GitHub, if it discovers a new repository, it registers it as a new resource_scopes row.
3. What is it returning in this query?
In get_required_ids, the query fetches the primary key ID (id) of that scope:

python
scope_rows = (
    supabase.table("resource_scopes")
    .select("id")
    .eq("external_resource_id", scope_external_id) # matches 'github:company/payment-service'
    .limit(1)
    .execute()
    .data
)
If it exists: It returns a list containing one dictionary with the ID:

python
scope_rows = [{"id": 8}]
(Line 30 grabs this value via scope_rows[0]["id"] which returns 8).

If it doesn't exist: It returns an empty list:

python
scope_rows = []
(This triggers the validation error on Line 28: "resource_scope not found").
